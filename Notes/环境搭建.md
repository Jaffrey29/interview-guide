## Docker 基本概念

可以用“宿舍楼”的比喻理解：

```text
电脑 = 一栋宿舍楼
Docker = 宿管系统，可以在电脑里开很多独立房间
镜像 image = 房间装修模板 / 预制菜包
容器 container = 按照镜像真正运行起来的房间
端口 port = 房间门牌号
端口映射 port mapping = 从电脑门牌转发到容器门牌的通道
数据卷 volume = 把电脑硬盘目录挂到容器里，防止容器删除后数据丢失
```

面试常问：

```text
镜像和容器的区别？
镜像是静态模板，容器是镜像运行后的实例。

为什么容器删了数据可能没了？
因为容器文件系统默认是临时的。重要数据要挂载 volume 或宿主机目录。

Docker 解决什么问题？
解决“我这里能跑，你那里跑不了”的环境一致性问题。
```

## 端口映射

端口映射格式：

```cmd
-p 宿主机端口:容器内端口
```

例如：

```cmd
-p 5433:5432
```

意思是：

```text
访问电脑的 localhost:5433
Docker 转发到容器内部的 5432
```

本次 PostgreSQL 为什么用 `5433:5432`：

```text
Windows 本机已有 PostgreSQL 占用了 localhost:5432
Docker 里的 PGvector PostgreSQL 内部仍然监听 5432
所以把电脑上的 5433 映射到容器里的 5432
```

最终关系：

```text
localhost:5432 -> Windows 本机 PostgreSQL
localhost:5433 -> Docker PGvector PostgreSQL
```

这也是为什么一开始 `CREATE EXTENSION vector;` 报错：当时 DataGrip 连到了 Windows 本机 PostgreSQL，而不是 Docker 里的 pgvector 镜像。

## PostgreSQL + PGvector

启动命令：


关键参数：

```text

-p 5433:5432

-e POSTGRES_PASSWORD=123456

-v C:\Users\J3axsco\docker\postgresql\data:/var/lib/postgresql/data
把数据库数据保存到 Windows 本地目录。

pgvector/pgvector:pg17
官方 PGvector 镜像，已经包含 PostgreSQL + vector 扩展文件。
```

DataGrip 连接：

```text

Port: 5433
User: postgres
Password: 123456
Database: interview_guide
```

启用扩展：

```sql

```

```sql

```

应看到：

面试常问：

## RustFS / S3 兼容对象存储

启动命令：

```cmd
docker run -d --name rustfs_local -p 9000:9000 -p 9001:9001 -v C:\Users\J3axsco\docker\rustfs\data:/data rustfs/rustfs:latest /data
```

端口：

```text
9000：S3 API 端口，后端程序上传/下载文件用
9001：Web Console 管理后台端口，浏览器访问用
```

管理后台：

```text
http://localhost:9001
用户名：rustfsadmin
密码：rustfsadmin
```

需要创建：

```text
Bucket: interview-guide
Access Key
Secret Key
```

在项目中对应配置为：

```yaml
app:
  storage:
    endpoint: ${APP_STORAGE_ENDPOINT:http://localhost:9000}
    access-key: ${APP_STORAGE_ACCESS_KEY:minioadmin}
    secret-key: ${APP_STORAGE_SECRET_KEY:minioadmin}
    bucket: ${APP_STORAGE_BUCKET:interview-guide}
    region: ${APP_STORAGE_REGION:us-east-1}
```

注意：这里不叫 `rustfs`，而叫 `app.storage`。因为项目代码面向的是 S3 兼容接口，RustFS、MinIO、云厂商对象存储都可以替换。

面试常问：

```text
对象存储适合存什么？
图片、PDF、Word、音频、视频等非结构化文件。

数据库和对象存储怎么分工？
数据库存结构化元数据，例如文件名、URL、状态、分析结果；
对象存储存真实文件内容。

S3 兼容是什么意思？
实现了 AWS S3 风格 API，应用可以用 AWS S3 SDK 访问，不绑定具体厂商。
```

## DeepSeek API

本次使用 DeepSeek 作为文字大模型 Provider。

项目配置位置：

```yaml
app:
  ai:
    default-provider: deepseek
    providers:
      deepseek:
        base-url: https://api.deepseek.com
        api-key: ${PROVIDER_DEEPSEEK_API_KEY:}
        model: ${PROVIDER_DEEPSEEK_MODEL:deepseek-v4-flash}
        supports-embedding: false
```

环境变量：

```cmd
setx PROVIDER_DEEPSEEK_API_KEY "你的DeepSeek API Key"
setx PROVIDER_DEEPSEEK_MODEL "deepseek-v4-flash"
```

安全原则：

```text
API Key 等同于密码。
不要发给别人。
不要截图公开。
不要提交到 GitHub。
如果完整暴露过，应删除并重新生成。
```

重要限制：

```text
DeepSeek 当前在项目配置中 supports-embedding: false
它可以负责聊天/分析
但知识库 RAG 的 embedding 可能还需要 DashScope、GLM 或本地 Ollama embedding 模型
```

语音面试提醒：

```text
项目里的 Qwen ASR/TTS 仍然使用 AI_BAILIAN_API_KEY
DeepSeek 不能替代 Qwen 实时语音服务
所以普通文字功能可以先跑，语音功能后面单独处理
```

## Gradle 和 gradlew

Gradle 是 Java 项目的构建工具，负责：

```text
下载 Java 依赖
编译 Java 代码
运行测试
打包 jar
启动 Spring Boot 后端
```

`gradlew.bat` 是 Gradle Wrapper：

```text
Wrapper = 项目自带的 Gradle 启动器
不用要求开发者电脑提前安装 Gradle
会使用项目指定的 Gradle 版本
避免“你电脑 Gradle 版本不对”的问题
```

启动后端：

```cmd
cd /d D:\XFG\JavaGuideSpringAI\interview-guide
gradlew.bat :app:bootRun
```

调试依赖下载：

```cmd
gradlew.bat :app:bootRun --info --no-daemon
```

成功标志：

```text
Tomcat started on port 8080
Started App in xx seconds
```

注意：

```text
bootRun 是长期运行任务。
后端活着时，它不会变成 100% 退出。
终端窗口就是后端运行窗口，不要关。
```

## Gradle 下载依赖很慢的问题

本次踩坑：

```text
Gradle 卡在 100% CONFIGURING
日志显示 maven.aliyun.com:443 Connection reset
```

原因：

```text
不是 Spring Boot 启动慢
而是 Gradle 在解析/下载依赖时，Maven 仓库连接被重置
UWA 校园网、防火墙、跨境链路、镜像源不稳定都可能导致
```

处理：

```text
调整 Gradle 仓库顺序，让 Maven Central 和腾讯云优先，阿里云兜底。
```

已改文件：

```text
interview-guide/settings.gradle
interview-guide/app/build.gradle
```

面试常问：

```text
什么是 Maven 仓库？
存放 Java 依赖包的远程仓库。

为什么第一次启动慢？
因为要下载大量 jar 包，之后本地缓存后会快很多。

什么是依赖缓存？
Gradle/Maven 会把下载过的依赖保存到本地，避免每次重新下载。
```

## application.yml 关键配置

数据库：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${POSTGRES_HOST:localhost}:${POSTGRES_PORT:5433}/interview_guide
    username: ${POSTGRES_USER:postgres}
    password: ${POSTGRES_PASSWORD:123456}
```

JPA 建表策略：

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: update
```

`ddl-auto` 说明：

```text
create：启动时删表重建，适合首次建表或测试，危险
update：保留现有数据，按实体变化增量更新，开发阶段常用
validate：只校验表结构，不修改
none/false：不让 Hibernate 管理表结构，生产环境更常见
```

本次曾经把 `ddl-auto` 改成 `create` 用于首次建表，后面已经改回 `update`。

面试常问：

```text
生产环境为什么不能随便用 ddl-auto: create？
因为会删除并重建表，导致数据丢失。
```

## 本项目的运行链路

简历上传分析的大致链路：

```text
前端上传简历
-> 后端 Controller 接收
-> Apache Tika 解析文件内容
-> RustFS 保存原始文件
-> PostgreSQL 保存元数据
-> Redis Stream 投递异步分析任务
-> DeepSeek 生成分析结果
-> PostgreSQL 保存分析结果
-> 前端查询并展示
```

知识库 RAG 的大致链路：

```text
上传知识库文档
-> RustFS 保存原始文档
-> 文档解析与切片
-> embedding 模型生成向量
-> PGvector 保存向量
-> 用户提问时检索相似文档片段
-> LLM 基于检索结果回答
```
