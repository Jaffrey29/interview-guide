#### 🚦 限流机制详解（核心重点）

```java
@RateLimit(dimension = RateLimit.Dimension.GLOBAL, count = 5)
@RateLimit(dimension = RateLimit.Dimension.IP, count = 5)
```

**为什么有两个 `@RateLimit`？**

* Java 8+ 支持 **可重复注解** （`@Repeatable`）
* 这两条规则是  **AND 关系** ，都要满足才能通过

**规则 1：全局限流**

```java
@RateLimit(dimension = RateLimit.Dimension.GLOBAL, count = 5)
```

* `dimension = GLOBAL`：对**所有用户**统计
* `count = 5`：每秒最多 5 次请求（默认 `interval=1, timeUnit=SECONDS`）
* **实际效果** ：整个系统每秒只能上传 5 份简历（防止 AI 接口被打爆）

**规则 2：IP 限流**

```java
@RateLimit(dimension = RateLimit.Dimension.IP, count = 5)
```

* `dimension = IP`：按**客户端 IP** 分别统计
* `count = 5`：每个 IP 每秒最多 5 次
* **实际效果** ：单个 IP 疯狂刷接口也被限制在每秒 5 次

**实战场景模拟：**

| 时间         | IP A 请求            | IP B 请求              | 全局总数 | 结果         |
| ------------ | -------------------- | ---------------------- | -------- | ------------ |
| 0.0s         | ✅ 第1次             | ✅ 第1次               | 2        | 都通过       |
| 0.2s         | ✅ 第2次             | ✅ 第2次               | 4        | 都通过       |
| 0.4s         | ✅ 第3次             | ❌ 第6次（超全局限流） | 6        | A通过，B被拒 |
| 0.6s         | ✅ 第4次             |                        | 7        | A通过        |
| 0.8s         | ✅ 第5次             |                        | 8        | A通过        |
| 1.0s         | ❌ 第6次（超IP限流） |                        | -        | A被拒        |
| 1.1s（新秒） | ✅ 第1次（计数重置） | ✅ 第1次               | 2        | 都通过       |

 **限流实现原理** （背后用了 Redis + Redisson）：

1. 每个维度对应一个 Redis key：
   * `rate_limit:GLOBAL:/api/resumes/upload`
   * `rate_limit:IP:192.168.1.100:/api/resumes/upload`
2. 使用 Redis 的  **RateLimiter** （令牌桶算法）
3. 超限时抛出 `RateLimitExceededException`，被全局异常处理器捕获返回 429

对比实体类（Entity）：

 **Entity（数据库映射）** ：

```java
@Entity
public class Resume {
```

```java
public class ResumeListItemDTO {

    private String originalFilename;
```

### **第 79-96 行：导出 PDF（特殊返回值）**

```java
@GetMapping("/api/resumes/{id}/export")
public ResponseEntity<byte[]> exportAnalysisPdf(@PathVariable Long id) {
    try {
        var result = historyService.exportAnalysisPdf(id);
        String filename = URLEncoder.encode(result.filename(), StandardCharsets.UTF_8);

        return ResponseEntity.ok()
            .header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename*=UTF-8''" + filename)
            .contentType(MediaType.APPLICATION_PDF)
            .body(result.pdfBytes());
    } catch (Exception e) {
        log.error("导出PDF失败: resumeId={}", id, e);
        return ResponseEntity.internalServerError().build();
    }
}
```

#### 🔥 为什么不用 `Result<byte[]>`？

**对比两种设计：**

 **❌ 错误设计** （用 Result 包文件）：

```java
public Result<byte[]> exportPdf(...) {
    return Result.success(pdfBytes);
}
```

前端收到：

```json
{
  "code": 200,
  "message": "success",
  "data": [37, 80, 68, 70, ...] // PDF 二进制转成了 JSON 数组！
}
```

 **问题** ：

* PDF 二进制变成 JSON 数组，体积膨胀 3-4 倍
* 浏览器无法直接识别为文件下载
* 前端还要转回二进制，麻烦且低效

 **✅ 正确设计** （用 ResponseEntity）：

```java
public ResponseEntity<byte[]> exportPdf(...) {
    return ResponseEntity.ok()
        .contentType(MediaType.APPLICATION_PDF)  // 告诉浏览器这是 PDF
        .header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename*=UTF-8''报告.pdf") // 触发下载
        .body(pdfBytes);  // 直接返回二进制
}
```

 **HTTP 响应（原始格式）** ：

```
HTTP/1.1 200 OK
Content-Type: application/pdf
Content-Disposition: attachment; filename*=UTF-8''%E6%8A%A5%E5%91%8A.pdf

%PDF-1.4
%âãÏÓ
1 0 obj
<<
/Type /Catalog
...（二进制数据）
```

 **浏览器行为** ：

* 看到 `Content-Disposition: attachment`，触发下载框
* 文件名显示为"报告.pdf"（URLEncoder 处理中文）
* 双击打开就是 PDF

#### 🔍 关键代码详解

**`URLEncoder.encode(result.filename(), StandardCharsets.UTF_8)`**

* 中文文件名"简历分析报告.pdf" → `%E7%AE%80%E5%8E%86%E5%88%86%E6%9E%90%E6%8A%A5%E5%91%8A.pdf`
* 防止文件名乱码

**`filename*=UTF-8''报告.pdf`** 语法

* RFC 2231 标准，支持非 ASCII 字符
* `*=UTF-8''` 表示后面是 UTF-8 编码的 URL 编码字符串

**异常处理**


* 返回 HTTP 500，body 为空

2. 用户点"重新分析"按钮
3. 后端重新投递任务到 Redis Stream
4. 消费者重新调用 AI 分析

## 🎯 核心知识点总结

### 1. 限流机制

| 维度   | 作用       | 实现              | 使用场景                   |
| ------ | ---------- | ----------------- | -------------------------- |
| GLOBAL | 全局总流量 | Redis RateLimiter | 保护下游（AI接口、数据库） |
| IP     | 单 IP 流量 | 按 IP 分桶        | 防止单个客户端滥用         |
| USER   | 单用户流量 | 按用户 ID 分桶    | VIP 用户配额管理           |

### 2. 返回值选择

| 场景         | 返回类型                   | 原因                       |
| ------------ | -------------------------- | -------------------------- |
| 普通业务数据 | `Result<T>`              | 统一格式，前端好处理       |
| 文件下载     | `ResponseEntity<byte[]>` | 二进制数据，触发浏览器下载 |
| 无返回数据   | `Result<Void>`           | 保持格式统一               |

### 3. DTO vs Entity

| 类型   | 作用         | 字段     | 使用位置              |
| ------ | ------------ | -------- | --------------------- |
| Entity | 数据库映射   | 完整字段 | Service/Repository 层 |
| DTO    | 接口数据传输 | 精简字段 | Controller 层返回     |

```text
再看 DTO 每个字段从哪里来
再看 Entity 如何映射数据库
最后写 Controller 调 Service，Service 组装 DTO
```

### 1. 为什么数据库里有很长的 JSON？

你在 Navicat 里看到的两个很长 JSON，主要对应 `resume_analyses` 表里的这两个字段：

```java
@Column(columnDefinition = "TEXT")
private String strengthsJson;

@Column(columnDefinition = "TEXT")
private String suggestionsJson;
```

它们在 `ResumeAnalysisEntity` 中，含义是：

```text
strengthsJson    AI 分析出来的简历优点列表
suggestionsJson  AI 分析出来的改进建议列表
```

作者这样设计，是因为 AI 返回的数据不是一个简单字符串，而是结构化列表。比如 `strengths` 适合表达成：

```json
[
  "熟悉 Java 和 Spring Boot",
  "有完整项目经验",
  "简历结构比较清晰"
]
```

`suggestions` 可能更复杂，像这样：

```json
[
  {
    "category": "项目经历",
    "problem": "缺少量化指标",
    "suggestion": "补充 QPS、用户量、性能提升百分比"
  }
]
```

如果严格关系型建模，也可以拆成三张表：

```text
resume_analyses
analysis_strengths
analysis_suggestions
```

但这个项目选择把它们直接存成 JSON 字符串，原因通常是：

```text
1. AI 返回结果本来就是 JSON，直接保存成本低。
2. strengths 和 suggestions 通常只用于展示，不经常按内部字段查询。
3. AI 结果结构以后可能变化，TEXT/JSON 比拆表更灵活。
4. 简历分析结果通常是一次生成、一整块读取，适合整体保存。
```

代价也很明确：

```text
1. Navicat 里看起来很长。
2. 数据库不方便按 JSON 内部字段做复杂查询。
3. Java 读取时要手动反序列化成 List。
```

所以，这种设计适合“展示型、结果型、结构可能变化”的 AI 数据，不适合“需要频繁筛选、统计、关联查询”的核心业务字段。

### 2. 保存 JSON 时，代码做了什么？

保存分析结果的核心方法在 `ResumePersistenceService.saveAnalysis(...)`：

```java
public ResumeAnalysisEntity saveAnalysis(ResumeEntity resume, ResumeAnalysisResponse analysis) {
    try {
        ResumeAnalysisEntity entity = resumeMapper.toAnalysisEntity(analysis);
        entity.setResume(resume);

        entity.setStrengthsJson(objectMapper.writeValueAsString(analysis.strengths()));
        entity.setSuggestionsJson(objectMapper.writeValueAsString(analysis.suggestions()));

        ResumeAnalysisEntity saved = analysisRepository.save(entity);
        return saved;
    } catch (JacksonException e) {
        throw new BusinessException(ErrorCode.RESUME_ANALYSIS_FAILED, "保存评测结果失败");
    }
}
```

逐行拆解：

```java
ResumeAnalysisEntity entity = resumeMapper.toAnalysisEntity(analysis);
```

把 AI 返回的 `ResumeAnalysisResponse` 转成数据库实体 `ResumeAnalysisEntity`。普通字段可以自动映射，例如：

```text
overallScore
contentScore
structureScore
skillMatchScore
expressionScore
projectScore
summary
```

```java
entity.setResume(resume);
```

建立“分析记录”和“简历”的关联，因为 `ResumeAnalysisEntity` 里有：

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "resume_id", nullable = false)
private ResumeEntity resume;
```

含义是：

```text
一份简历可以有多条分析记录。
一条分析记录只属于一份简历。
数据库里通过 resume_id 关联。
```

```java
objectMapper.writeValueAsString(analysis.strengths())
```

把 Java 里的 `List<String>` 转成 JSON 字符串，保存进 `strengthsJson`。

```java
objectMapper.writeValueAsString(analysis.suggestions())
```

把 Java 里的建议列表转成 JSON 字符串，保存进 `suggestionsJson`。

最后：

```java
analysisRepository.save(entity);
```

通过 Spring Data JPA 写入数据库表 `resume_analyses`。

### 3. 读取 JSON 时，代码做了什么？

详情接口希望返回的是：

```java
List<String> strengths
List<Object> suggestions
```

但数据库里存的是：

```java
String strengthsJson
String suggestionsJson
```

所以读取时要把 JSON 字符串解析回 Java List。代码在 `ResumeHistoryService`：

```java
private List<String> extractStrengths(ResumeAnalysisEntity entity) {
    try {
        if (entity.getStrengthsJson() != null) {
            return objectMapper.readValue(
                entity.getStrengthsJson(),
                new TypeReference<>() {}
            );
        }
    } catch (JacksonException e) {
        log.error("解析 strengths JSON 失败", e);
    }
    return List.of();
}
```

这段逻辑是：

```text
如果 strengthsJson 不为空：
    用 objectMapper.readValue 把 JSON 字符串转成 List<String>
如果为空或解析失败：
    返回空列表 List.of()
```

为什么返回 `List.of()`，而不是 `null`？

```text
1. 前端更好处理，空数组比 null 更安全。
2. 避免后续代码 NullPointerException。
3. 表示“没有数据”，而不是“系统崩了”。
```

`suggestions` 同理：

```java
private List<Object> extractSuggestions(ResumeAnalysisEntity entity) {
    try {
        if (entity.getSuggestionsJson() != null) {
            return objectMapper.readValue(
                entity.getSuggestionsJson(),
                new TypeReference<>() {}
            );
        }
    } catch (JacksonException e) {
        log.error("解析 suggestions JSON 失败", e);
    }
    return List.of();
}
```

这里用 `List<Object>`，说明建议的数据结构可能比字符串更复杂，可能是对象列表。作者用 `Object` 保留灵活性。

### 4. ResumeController 的职责

`ResumeController` 是简历模块的 HTTP 入口。它不写数据库查询、不写 AI 分析细节、不手动解析 JSON。

它只负责：

```text
1. 声明 URL。
2. 接收 HTTP 参数。
3. 调用对应 Service。
4. 把 Service 返回值包装成统一响应。
```

类定义：

```java
@Slf4j
@RestController
@RequiredArgsConstructor
@Tag(name = "简历管理", description = "简历上传、分析、导出与删除")
public class ResumeController {

    private final ResumeUploadService uploadService;
    private final ResumeDeleteService deleteService;
    private final ResumeHistoryService historyService;
}
```

注解解释：

```java
@Slf4j
```

Lombok 自动生成 `log` 日志对象，所以可以直接写：

```java
log.error("导出PDF失败: resumeId={}", id, e);
```

```java
@RestController
```

Spring MVC 注解，表示这是 REST 接口类。方法返回的 Java 对象会自动序列化成 JSON。

```java
@RequiredArgsConstructor
```

Lombok 给所有 `final` 字段生成构造函数，等价于：

```java
public ResumeController(
    ResumeUploadService uploadService,
    ResumeDeleteService deleteService,
    ResumeHistoryService historyService
) {
    this.uploadService = uploadService;
    this.deleteService = deleteService;
    this.historyService = historyService;
}
```

Spring 创建 Controller 时，会自动把这三个 Service 注入进来。

```java
@Tag(...)
```

Swagger/OpenAPI 文档注解，用于接口文档分组，不影响业务逻辑。

三个 `private final` 字段：

```java
private final ResumeUploadService uploadService;
private final ResumeDeleteService deleteService;
private final ResumeHistoryService historyService;
```

含义是：

```text
private：只允许当前类内部使用。
final：对象创建后不能换成另一个 Service。
Service：具体业务逻辑交给业务层处理。
```

分工：

```text
uploadService   上传、解析、重新分析
deleteService   删除简历以及关联数据
historyService  查询列表、详情、导出 PDF
```

### 5. 上传简历 uploadAndAnalyze

```java
@PostMapping(value = "/api/resumes/upload", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
@RateLimit(dimension = RateLimit.Dimension.GLOBAL, count = 5)
@RateLimit(dimension = RateLimit.Dimension.IP, count = 5)
public Result<Map<String, Object>> uploadAndAnalyze(@RequestParam("file") MultipartFile file) {
    Map<String, Object> result = uploadService.uploadAndAnalyze(file);
    boolean isDuplicate = (Boolean) result.get("duplicate");
    if (isDuplicate) {
        return Result.success("检测到相同简历，已返回历史分析结果", result);
    }
    return Result.success(result);
}
```

复现前先判断：

```text
这是文件上传，所以用 POST。
请求体是 multipart/form-data。
文件参数来自表单字段 file。
上传细节交给 uploadService。
返回值统一包装成 Result。
```

`@PostMapping` 声明接口：

```text
POST /api/resumes/upload
Content-Type: multipart/form-data
```

`@RequestParam("file") MultipartFile file` 表示从表单字段 `file` 中取上传文件。前端必须这样传：

```javascript
const formData = new FormData();
formData.append("file", file);
```

`Result<Map<String, Object>>` 拆开看：

```text
Result<T>               统一响应外壳
Map<String, Object>     data 里的实际内容
```

这里用 Map，是因为上传返回内容比较动态，可能有 `resumeId`、`duplicate`、`analyzeStatus`、`filename`、`score`。更严格的写法也可以定义 `ResumeUploadResultDTO`。

### 8. 导出 PDF exportAnalysisPdf

```java
@GetMapping("/api/resumes/{id}/export")
public ResponseEntity<byte[]> exportAnalysisPdf(@PathVariable Long id) {
    try {
        var result = historyService.exportAnalysisPdf(id);
        String filename = URLEncoder.encode(result.filename(), StandardCharsets.UTF_8);

        return ResponseEntity.ok()
            .header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename*=UTF-8''" + filename)
            .contentType(MediaType.APPLICATION_PDF)
            .body(result.pdfBytes());
    } catch (Exception e) {
        log.error("导出PDF失败: resumeId={}", id, e);
        return ResponseEntity.internalServerError().build();
    }
}
```

这里不用 `Result<byte[]>`，因为 PDF 是二进制文件，不是普通 JSON。`ResponseEntity<byte[]>` 可以控制 HTTP 状态码、响应头、Content-Type 和响应体二进制内容。

关键代码：

```java
.contentType(MediaType.APPLICATION_PDF)
```

告诉浏览器这是 PDF。

```java
.header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename*=UTF-8''" + filename)
```

告诉浏览器以附件形式下载。

```java
URLEncoder.encode(result.filename(), StandardCharsets.UTF_8)
```

处理中文文件名，避免下载时乱码。

```java
.body(result.pdfBytes())
```

返回 PDF 二进制内容。

### 10. 重新分析 reanalyze

```java
@PostMapping("/api/resumes/{id}/reanalyze")
```

因为重新分析会消耗 AI Token 和后台资源。

### 12. 自己复现 ResumeController 的步骤

不要背代码，按这个顺序推：

```text
1. 确定业务动作：上传、列表、详情、导出、删除、重新分析、健康检查。
2. 选择 HTTP 方法：POST、GET、GET、GET、DELETE、POST、GET。
3. 设计路径：/api/resumes/upload、/api/resumes、/api/resumes/{id}/detail 等。
4. 确定参数来源：@RequestParam 文件、@PathVariable id、无参数。
5. 确定返回值：Result<T> 或 ResponseEntity<byte[]>。
6. 确定调用哪个 Service：uploadService、deleteService、historyService。
```

核心骨架：

```java
@RestController
@RequiredArgsConstructor
public class ResumeController {

    private final ResumeUploadService uploadService;
    private final ResumeDeleteService deleteService;
    private final ResumeHistoryService historyService;

    @PostMapping(value = "/api/resumes/upload", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
    public Result<Map<String, Object>> uploadAndAnalyze(@RequestParam("file") MultipartFile file) {
        Map<String, Object> result = uploadService.uploadAndAnalyze(file);
        return Result.success(result);
    }

    @GetMapping("/api/resumes")
    public Result<List<ResumeListItemDTO>> getAllResumes() {
        return Result.success(historyService.getAllResumes());
    }

    @GetMapping("/api/resumes/{id}/detail")
    public Result<ResumeDetailDTO> getResumeDetail(@PathVariable Long id) {
        return Result.success(historyService.getResumeDetail(id));
    }

    @DeleteMapping("/api/resumes/{id}")
    public Result<Void> deleteResume(@PathVariable Long id) {
        deleteService.deleteResume(id);
        return Result.success(null);
    }
}
```

之后再补上限流注解、PDF 导出、重复上传提示、健康检查、日志、Swagger 注解。

## Redis 在本项目里的实战用法：从会话缓存到异步任务

这一节专门解决一个问题：Redis 在项目里到底怎么用，怎么从代码看出它存了什么，自己写项目时怎么复现。

先记住一句话：

```text
Redis 不是用来替代数据库的。
Redis 适合存“快、短期、会频繁变化、可以过期、可以重建”的数据。
数据库适合存“长期、可靠、需要查询和留痕”的数据。
```

在这个项目里，Redis 主要有三类用法：

```text
1. 面试会话缓存：interview:session:{sessionId}
2. 简历/面试异步任务队列：Redis Stream
3. 接口限流：ratelimit:{class:method}:...
```

你现在在 Another Redis Desktop 里看到的 key，就是这些代码运行后写进去的。

### 1. Redis 是怎么接入项目的？

Gradle 里引入了 Redisson：

```gradle
implementation "org.redisson:redisson-spring-boot-starter:${libs.versions.redisson.get()}"
```

配置在 `application.yml`：

```yaml
spring:
  redis:
    redisson:
      config: |
        singleServerConfig:
          address: "redis://${REDIS_HOST:localhost}:${REDIS_PORT:6379}"
          database: 0
```

含义：

```text
使用 Redis 单机模式
连接 localhost:6379
使用 Redis database 0
```

Redisson starter 启动时会根据这个配置创建：

```java
RedissonClient
```

然后项目自己的 `RedisService` 注入它：

```java
@Service
@RequiredArgsConstructor
public class RedisService {
    private final RedissonClient redissonClient;
}
```

所以完整链路是：

```text
application.yml
  -> redisson-spring-boot-starter
      -> RedissonClient
          -> RedisService
              -> InterviewSessionCache / StreamProducer / RateLimitAspect
```

编译器只检查类型，真正连接 Redis 是运行时由 Spring + Redisson 配置完成的。

### 2. RedisService 是项目里的 Redis 封装层

项目没有让业务代码到处直接写：

```java
redissonClient.getBucket(...)
```

而是封装成了：

```java
RedisService
```

常见方法：

```java
public <T> void set(String key, T value, Duration ttl) {
    RBucket<T> bucket = redissonClient.getBucket(key);
    bucket.set(value, ttl);
}

public <T> T get(String key) {
    RBucket<T> bucket = redissonClient.getBucket(key);
    return bucket.get();
}

public boolean delete(String key) {
    return redissonClient.getBucket(key).delete();
}

public boolean expire(String key, Duration ttl) {
    return redissonClient.getBucket(key).expire(ttl);
}
```

这里的 `RBucket` 可以理解成 Redis 里的普通 key-value。

例如：

```java
redisService.set("interview:session:abc", cachedSession, Duration.ofHours(24));
```

等价于：

```text
在 Redis 里保存一个 key：
key   = interview:session:abc
value = cachedSession 对象
TTL   = 24 小时
```

RedisService 的作用是：

```text
1. 屏蔽 Redisson 细节。
2. 让业务层只关心 set/get/delete/streamAdd。
3. 统一管理 Redis 操作。
```

以后你自己写项目时，不建议在业务 Service 里到处散落 Redisson 原生 API。更好的方式也是先封装一个 `RedisService`。

### 3. 面试会话为什么要存 Redis？

面试会话是典型适合 Redis 的数据。

一场面试进行中，会频繁变化：

```text
当前第几题
题目列表
用户已经答了哪些题
会话状态 CREATED / IN_PROGRESS / COMPLETED
```

这些数据有两个特点：

```text
1. 读写频繁。
2. 不是永久数据，过一段时间可以过期。
```

所以它适合放 Redis。

但数据库仍然要保存一份长期记录：

```text
interview_sessions
interview_answers
```

所以本项目采用的是：

```text
Redis：保存正在进行的会话状态，快。
PostgreSQL：保存长期历史记录，可靠。
```

这就是实战里常见的“缓存 + 数据库”组合。

### 4. InterviewSessionCache 管理哪些 key？

代码在：

```text
InterviewSessionCache
```

它定义了两个 key 前缀：

```java
private static final String SESSION_KEY_PREFIX = "interview:session:";
private static final String RESUME_SESSION_KEY_PREFIX = "interview:resume:";
```

所以你在 Another Redis Desktop 里会看到类似：

```text
interview:session:2d67c887b4be44fb
interview:resume:2
```

两个 key 的含义不同。

#### 4.1 interview:session:{sessionId}

这个 key 存一整场面试会话。

代码：

```java
private String buildSessionKey(String sessionId) {
    return SESSION_KEY_PREFIX + sessionId;
}
```

如果：

```text
sessionId = 2d67c887b4be44fb
```

那么 Redis key 就是：

```text
interview:session:2d67c887b4be44fb
```

value 是 `CachedSession`：

```java
public static class CachedSession implements Serializable {
    private String sessionId;
    private String resumeText;
    private Long resumeId;
    private String questionsJson;
    private int currentIndex;
    private SessionStatus status;
}
```

字段解释：

```text
sessionId      当前面试会话 ID
resumeText     简历文本
resumeId       关联的简历 ID
questionsJson  题目列表序列化后的 JSON
currentIndex   当前进行到第几题
status         会话状态
```

所以这个 Redis key 存的是“当前面试现场”。

#### 4.2 interview:resume:{resumeId}

这个 key 用于根据简历 ID 找未完成会话。

代码：

```java
private String buildResumeSessionKey(Long resumeId) {
    return RESUME_SESSION_KEY_PREFIX + resumeId;
}
```

如果：

```text
resumeId = 2
```

那么 Redis key 是：

```text
interview:resume:2
```

value 通常是：

```text
某个 sessionId
```

作用是：

```text
这份简历当前有没有未完成面试？
如果有，未完成面试的 sessionId 是什么？
```

这就是为什么 `createSession` 一开始可以检查未完成会话。

### 5. saveSession 是怎么写入 Redis 的？

`InterviewSessionService.createSession(...)` 里调用：

```java
sessionCache.saveSession(
    sessionId,
    request.resumeText() != null ? request.resumeText() : "",
    request.resumeId(),
    questions,
    0,
    SessionStatus.CREATED
);
```

进入 `InterviewSessionCache.saveSession(...)`：

```java
public void saveSession(String sessionId, String resumeText, Long resumeId,
                       List<InterviewQuestionDTO> questions, int currentIndex,
                       SessionStatus status) {
    String key = buildSessionKey(sessionId);
    CachedSession cachedSession = new CachedSession(
        sessionId, resumeText, resumeId, questions, currentIndex, status, objectMapper
    );

    redisService.set(key, cachedSession, SESSION_TTL);

    if (resumeId != null && isUnfinishedStatus(status)) {
        saveResumeSessionMapping(resumeId, sessionId);
    }
}
```

逐行解释：

```java
String key = buildSessionKey(sessionId);
```

生成 Redis key：

```text
interview:session:{sessionId}
```

```java
CachedSession cachedSession = new CachedSession(...);
```

把 Java 参数包装成一个缓存对象。

```java
redisService.set(key, cachedSession, SESSION_TTL);
```

真正写入 Redis，并设置过期时间。

`SESSION_TTL` 是：

```java
private static final Duration SESSION_TTL = Duration.ofHours(24);
```

所以会话缓存会保留 24 小时。

```java
if (resumeId != null && isUnfinishedStatus(status)) {
    saveResumeSessionMapping(resumeId, sessionId);
}
```

如果这场面试绑定了简历，并且状态还是未完成，就额外保存一份映射：

```text
interview:resume:{resumeId} -> sessionId
```

这样下次创建面试时，可以快速判断这份简历是否已经有未完成会话。

### 6. questionsJson 为什么又是 JSON？

`CachedSession` 里有：

```java
private String questionsJson;
```

构造函数里：

```java
this.questionsJson = objectMapper.writeValueAsString(questions);
```

意思是：

```text
把 List<InterviewQuestionDTO> 转成 JSON 字符串后存入 Redis。
```

读取时：

```java
public List<InterviewQuestionDTO> getQuestions(ObjectMapper objectMapper) {
    return objectMapper.readValue(questionsJson, new TypeReference<>() {});
}
```

意思是：

```text
把 Redis 里的 questionsJson 再转回 List<InterviewQuestionDTO>。
```

为什么不直接把 List 原样存 Redis？

因为 Redis 最终存的是二进制/字符串数据。对象必须经过序列化。项目选择在 `CachedSession` 内部明确把题目列表转成 JSON 字符串，这样结构更清楚，也便于排查。

### 7. getSession 如何从 Redis 读取？

代码：

```java
public Optional<CachedSession> getSession(String sessionId) {
    String key = buildSessionKey(sessionId);
    CachedSession session = redisService.get(key);
    if (session != null) {
        return Optional.of(session);
    }
    return Optional.empty();
}
```

逻辑：

```text
1. 根据 sessionId 拼出 key。
2. 调 redisService.get(key)。
3. 如果 Redis 里有值，返回 Optional.of(session)。
4. 如果没有，返回 Optional.empty()。
```

所以在 `InterviewSessionService.getSession(...)` 里会先查 Redis：

```java
Optional<CachedSession> cachedOpt = sessionCache.getSession(sessionId);
if (cachedOpt.isPresent()) {
    return toDTO(cachedOpt.get());
}
```

如果 Redis 没有，再从数据库恢复：

```java
CachedSession restoredSession = restoreSessionFromDatabase(sessionId);
```

这就是常见的缓存策略：

```text
先查 Redis
Redis 没有，再查数据库
查到后再写回 Redis
```

### 8. findUnfinishedSessionId 怎么用 Redis 快速找未完成会话？

代码：

```java
public Optional<String> findUnfinishedSessionId(Long resumeId) {
    String key = buildResumeSessionKey(resumeId);
    String sessionId = redisService.get(key);
    if (sessionId != null) {
        Optional<CachedSession> sessionOpt = getSession(sessionId);
        if (sessionOpt.isPresent() && isUnfinishedStatus(sessionOpt.get().getStatus())) {
            return Optional.of(sessionId);
        } else {
            redisService.delete(key);
        }
    }
    return Optional.empty();
}
```

逻辑：

```text
1. 用 resumeId 拼 key：interview:resume:{resumeId}
2. 从 Redis 读取 sessionId
3. 如果能读到，再检查 interview:session:{sessionId} 是否存在
4. 如果会话存在并且未完成，返回这个 sessionId
5. 如果会话不存在或已经完成，删除这个映射 key
```

为什么要二次检查？

因为 `interview:resume:2` 只是一个映射，可能出现映射还在，但真正的 session key 已过期或状态已完成。

所以代码必须校验：

```java
sessionOpt.isPresent() && isUnfinishedStatus(sessionOpt.get().getStatus())
```

这个设计很实用：Redis 里存索引 key 时，要考虑索引和真实数据不一致的情况。

### 9. 更新当前题目和状态时，Redis 怎么变？

提交答案时，`InterviewSessionService.submitAnswer(...)` 会做：

```java
sessionCache.updateQuestions(request.sessionId(), questions);
sessionCache.updateCurrentIndex(request.sessionId(), newIndex);
```

如果答完了，还会：

```java
sessionCache.updateSessionStatus(request.sessionId(), SessionStatus.COMPLETED);
```

`updateCurrentIndex`：

```java
public void updateCurrentIndex(String sessionId, int currentIndex) {
    getSession(sessionId).ifPresent(session -> {
        session.setCurrentIndex(currentIndex);
        String key = buildSessionKey(sessionId);
        redisService.set(key, session, SESSION_TTL);
    });
}
```

它的做法是：

```text
1. 从 Redis 读出 CachedSession。
2. 修改 currentIndex。
3. 整个对象重新 set 回 Redis。
4. 重新刷新 TTL。
```

`updateSessionStatus` 也是类似：

```java
session.setStatus(status);
redisService.set(key, session, SESSION_TTL);
```

如果状态变成完成，就删掉简历到未完成会话的映射：

```java
if (!isUnfinishedStatus(status) && session.getResumeId() != null) {
    removeResumeSessionMapping(session.getResumeId(), sessionId);
}
```

因为完成后，这份简历不应该再被认为“有未完成面试”。

### 10. 简历管理里 Redis 用在哪里？

简历管理中最重要的 Redis 用法不是缓存简历详情，而是“异步分析任务队列”。

上传简历后，后端不会一直阻塞等待 AI 分析完成，而是：

```text
1. 先保存简历记录到数据库。
2. 把分析任务投递到 Redis Stream。
3. 后台消费者从 Stream 取任务。
4. 消费者调用 AI 分析。
5. 分析完成后把结果写回数据库。
```

生产者是：

```text
AnalyzeStreamProducer
```

关键代码：

```java
public void sendAnalyzeTask(Long resumeId, String content) {
    sendTask(new AnalyzeTaskPayload(resumeId, content));
}
```

它继承：

```java
AbstractStreamProducer
```

最终调用：

```java
redisService.streamAdd(
    streamKey(),
    buildMessage(payload),
    AsyncTaskStreamConstants.STREAM_MAX_LEN
);
```

简历分析 Stream key 是：

```java
public static final String RESUME_ANALYZE_STREAM_KEY = "resume:analyze:stream";
```

消息内容：

```java
Map.of(
    "resumeId", payload.resumeId().toString(),
    "content", payload.content(),
    "retryCount", "0"
)
```

所以你在 Redis 可视化工具里可能看到：

```text
resume:analyze:stream
```

里面的消息字段大概是：

```text
resumeId    简历 ID
content     简历文本
retryCount  重试次数
```

注意：如果消费者很快处理完并 ack，Stream 里可能只保留有限消息，或者 pending 为空。

### 11. InterviewController 相关 Redis Stream：面试评估任务

面试答完后，项目不会在用户提交最后一题时同步生成报告，而是投递异步评估任务。

在 `InterviewSessionService.submitAnswer(...)` 中：

```java
if (newStatus == SessionStatus.COMPLETED) {
    sessionCache.updateSessionStatus(request.sessionId(), SessionStatus.COMPLETED);
    enqueueEvaluationTask(request.sessionId());
}
```

`enqueueEvaluationTask`：

```java
private void enqueueEvaluationTask(String sessionId) {
    persistenceService.updateEvaluateStatus(sessionId, AsyncTaskStatus.PENDING, null);
    evaluateStreamProducer.sendEvaluateTask(sessionId);
}
```

这里做了两件事：

```text
1. 数据库里把 evaluateStatus 改成 PENDING。
2. 把 sessionId 投递到 Redis Stream。
```

生产者是：

```text
EvaluateStreamProducer
```

Stream key 是：

```java
public static final String INTERVIEW_EVALUATE_STREAM_KEY = "interview:evaluate:stream";
```

消息内容：

```java
Map.of(
    "sessionId", sessionId,
    "retryCount", "0"
)
```

所以你可能在 Redis 里看到：

```text
interview:evaluate:stream
```

这就是“面试评估任务队列”。

### 12. Redis Stream 的生产者和消费者怎么配合？

生产者负责写消息：

```text
AnalyzeStreamProducer       写 resume:analyze:stream
EvaluateStreamProducer      写 interview:evaluate:stream
```

消费者负责读消息：

```text
AnalyzeStreamConsumer       读 resume:analyze:stream
EvaluateStreamConsumer      读 interview:evaluate:stream
```

公共逻辑在：

```text
AbstractStreamProducer
AbstractStreamConsumer
```

生产者模板：

```java
protected void sendTask(T payload) {
    String messageId = redisService.streamAdd(
        streamKey(),
        buildMessage(payload),
        AsyncTaskStreamConstants.STREAM_MAX_LEN
    );
}
```

消费者模板启动时会：

```java
redisService.createStreamGroup(streamKey(), groupName());
```

然后循环消费：

```java
redisService.streamConsumeMessages(
    streamKey(),
    groupName(),
    consumerName,
    BATCH_SIZE,
    POLL_INTERVAL_MS,
    PENDING_IDLE_TIMEOUT_MS,
    PENDING_CLAIM_BATCH_SIZE,
    this::processMessage
);
```

处理成功后 ack：

```java
redisService.streamAck(streamKey(), groupName(), messageId);
```

所以 Redis Stream 的流程是：

```text
Controller/Service 触发任务
-> Producer 写入 Redis Stream
-> Consumer 后台线程读取 Stream
-> 执行业务
-> 成功后 ack
-> 失败则重试或标记 FAILED
```

### 14. 限流为什么也用 Redis？

Controller 中有：

```java
@RateLimit(dimension = RateLimit.Dimension.GLOBAL, count = 5)
@RateLimit(dimension = RateLimit.Dimension.IP, count = 5)
```

比如上传简历接口：

```java
@PostMapping(value = "/api/resumes/upload", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
@RateLimit(dimension = RateLimit.Dimension.GLOBAL, count = 5)
@RateLimit(dimension = RateLimit.Dimension.IP, count = 5)
public Result<Map<String, Object>> uploadAndAnalyze(...)
```

这些注解会被 `RateLimitAspect` 拦截。

它会生成 Redis key：

```java
String hashTag = "{" + className + ":" + methodName + "}";
String keyPrefix = "ratelimit:" + hashTag;
```

不同维度生成不同 key：

```java
GLOBAL -> ratelimit:{className:methodName}:global
IP     -> ratelimit:{className:methodName}:ip:{clientIp}
USER   -> ratelimit:{className:methodName}:user:{userId}
```

为什么限流适合 Redis？

```text
1. 计数变化非常频繁。
2. 需要过期窗口，比如每秒、每分钟。
3. 多个后端实例部署时，大家要共享同一份计数。
```

如果用 Java 内存计数，多个后端实例之间不共享。用 Redis，就可以全局共享。

### 15. 什么时候该用 Redis？从这个项目总结

适合用 Redis 的场景：

```text
1. 临时会话状态：面试当前第几题、题目列表、进行中状态。
2. 映射关系：resumeId -> 未完成 sessionId。
3. 异步任务队列：简历分析、面试评估。
4. 限流计数：每秒最多几次请求。
5. 分布式锁：多个实例抢同一个任务或资源时。
```

不适合只放 Redis 的场景：

```text
1. 简历主记录。
2. 面试历史。
3. 用户答案。
4. AI 分析报告。
5. 需要长期保留和复杂查询的数据。
```

这些应该放数据库。

所以本项目的设计是：

```text
Redis 存临时状态、队列、计数。
PostgreSQL 存最终业务数据。
RustFS 存原始文件。
```

### 16. 自己复现 Redis 会话缓存怎么写？

如果你要从零复现 `InterviewSessionCache`，不要先背代码。先写需求：

```text
我要缓存一场面试会话。
需要通过 sessionId 找到它。
需要通过 resumeId 找到未完成会话。
需要设置 24 小时过期。
需要能更新题目、当前索引、状态。
```

然后设计 key：

```text
interview:session:{sessionId}  -> CachedSession
interview:resume:{resumeId}    -> sessionId
```

然后设计 value：

```java
class CachedSession {
    private String sessionId;
    private String resumeText;
    private Long resumeId;
    private String questionsJson;
    private int currentIndex;
    private SessionStatus status;
}
```

然后写核心方法：

```java
public void saveSession(...) {
    String key = "interview:session:" + sessionId;
    CachedSession session = new CachedSession(...);
    redisService.set(key, session, Duration.ofHours(24));

    if (resumeId != null && isUnfinishedStatus(status)) {
        redisService.set("interview:resume:" + resumeId, sessionId, Duration.ofHours(24));
    }
}
```

读取：

```java
public Optional<CachedSession> getSession(String sessionId) {
    CachedSession session = redisService.get("interview:session:" + sessionId);
    return session != null ? Optional.of(session) : Optional.empty();
}
```

更新当前题：

```java
public void updateCurrentIndex(String sessionId, int index) {
    getSession(sessionId).ifPresent(session -> {
        session.setCurrentIndex(index);
        redisService.set("interview:session:" + sessionId, session, Duration.ofHours(24));
    });
}
```

这就是最小可复现版本。

### 17. 自己复现 Redis Stream 异步任务怎么写？

如果你要异步处理简历分析，先写需求：

```text
上传接口不能一直等 AI。
上传成功后投递一个任务。
后台慢慢消费任务。
失败可以重试。
最终结果写回数据库。
```

设计 Stream key：

```text
resume:analyze:stream
```

设计消息字段：

```text
resumeId
content
retryCount
```

生产者核心代码：

```java
public void sendAnalyzeTask(Long resumeId, String content) {
    redisService.streamAdd(
        "resume:analyze:stream",
        Map.of(
            "resumeId", resumeId.toString(),
            "content", content,
            "retryCount", "0"
        ),
        1000
    );
}
```

消费者核心流程：

```text
1. 创建 consumer group。
2. 循环读取 stream 消息。
3. 解析 resumeId 和 content。
4. 调用 AI 分析。
5. 写数据库。
6. ack 消息。
7. 失败则重新投递或标记失败。
```

本项目已经把这些通用逻辑抽到了：

```text
AbstractStreamProducer
AbstractStreamConsumer
```

所以具体业务类只需要告诉模板：

```text
Stream key 是什么
消息字段是什么
怎么处理业务
失败怎么更新状态
```

### 18. 在 Another Redis Desktop 里怎么看？

连接配置：

```text

```


你真正要掌握的不是 Redis 命令，而是这套判断方法：

## Controller 与 Service 的关系：为什么 Service 里面还能继续套 Service？

你现在看到的现象是：

```text
Controller 依赖 Service
Service 里面又依赖其他 Service / Cache / Producer / Registry / ObjectMapper
继续 Ctrl + 左键还能进入更细的类
```

这不是“无限套娃”，而是后端项目里非常常见的职责拆分。一个项目能不能复现，关键不是背类名，而是能判断：

```text
这个类属于哪一层？
它负责接 HTTP、编排业务、访问数据库、调用 AI，还是做缓存/消息？
```

### 1. Controller 里到底放什么？

Controller 不是只放“方法名和返回值”。更准确地说，Controller 是 HTTP 适配层。

它负责：

```text
1. 声明接口路径：@GetMapping、@PostMapping、@DeleteMapping
2. 接收请求参数：@RequestParam、@PathVariable、@RequestBody
3. 做非常轻量的请求级判断
4. 调用 Service
5. 把 Service 结果包装成 HTTP 响应
```

它通常不负责：

```text
1. 直接查数据库
2. 直接写 Repository
3. 直接拼复杂 DTO
4. 直接调用 AI 模型
5. 直接操作 Redis 队列
6. 写大段业务流程
```

以 `ResumeController` 为例：

```java
@GetMapping("/api/resumes")
public Result<List<ResumeListItemDTO>> getAllResumes() {
    List<ResumeListItemDTO> resumes = historyService.getAllResumes();
    return Result.success(resumes);
}
```

Controller 做的事情只有：

```text
收到 GET /api/resumes
调用 historyService.getAllResumes()
用 Result.success(...) 包起来返回
```

真正的数据库查询、最新分数查询、面试次数统计、DTO 组装，都在 `ResumeHistoryService`。

所以一句话：

```text
Controller 管“这个 HTTP 请求交给谁处理”。
Service 管“这件业务到底怎么完成”。
```

### 2. Service 里面为什么还能依赖其他 Service？

因为一个业务动作通常不是一个步骤。

比如“创建一场面试会话”这件事，不只是 `new InterviewSessionDTO`，它至少包括：

```text
1. 判断是否已经有未完成会话
2. 生成 sessionId
3. 读取技能方向和难度
4. 查询历史问题，避免重复出题
5. 调用 AI 生成题目
6. 保存会话到 Redis 缓存
7. 保存会话到数据库
8. 返回 DTO 给前端
```

如果这些都写在 Controller，就会变成几百行；如果都写在一个 Service，也会越来越难维护。所以 `InterviewSessionService` 继续依赖更细的组件：

```java
private final InterviewQuestionService questionService;
private final AnswerEvaluationService evaluationService;
private final InterviewPersistenceService persistenceService;
private final InterviewSessionCache sessionCache;
private final ObjectMapper objectMapper;
private final EvaluateStreamProducer evaluateStreamProducer;
private final LlmProviderRegistry llmProviderRegistry;
```

它们都是 `InterviewSessionService` 的成员变量，也就是这个 Service 完成业务流程所需要的协作者。

注意：它们不全都是“Service”。

```text
InterviewQuestionService      负责生成面试题
AnswerEvaluationService       负责评估面试回答、生成报告
InterviewPersistenceService   负责面试会话、答案、报告的数据库持久化
InterviewSessionCache         负责 Redis 缓存会话状态
ObjectMapper                  负责 JSON 字符串和 Java 对象互转
EvaluateStreamProducer        负责把评估任务投递到 Redis Stream
LlmProviderRegistry           负责选择和创建 LLM ChatClient
```

所以它们不是一堆“前台”，而是一组分工明确的“工具/岗位/下游服务”。

### 3. private final 在这里是什么意思？

比如：

```java
private final InterviewQuestionService questionService;
```

拆开理解：

```text
private：这个依赖只给 InterviewSessionService 内部使用。
final：构造完成后，questionService 不能被换成别的对象。
InterviewQuestionService：负责出题这个子任务。
```

`InterviewSessionService` 上有：

```java
@RequiredArgsConstructor
```

所以 Lombok 会生成构造函数：

```java
public InterviewSessionService(
    InterviewQuestionService questionService,
    AnswerEvaluationService evaluationService,
    InterviewPersistenceService persistenceService,
    InterviewSessionCache sessionCache,
    ObjectMapper objectMapper,
    EvaluateStreamProducer evaluateStreamProducer,
    LlmProviderRegistry llmProviderRegistry
) {
    this.questionService = questionService;
    this.evaluationService = evaluationService;
    this.persistenceService = persistenceService;
    this.sessionCache = sessionCache;
    this.objectMapper = objectMapper;
    this.evaluateStreamProducer = evaluateStreamProducer;
    this.llmProviderRegistry = llmProviderRegistry;
}
```

Spring 启动时会从容器里找这些 Bean，然后注入进来。这就是构造器注入。

为什么推荐这种写法？

```text
1. 依赖关系一眼可见。
2. 对象创建后依赖不会被替换。
3. 单元测试时可以手动传入 mock 对象。
4. 避免字段为 null。
```

### 4. InterviewSessionService 是什么角色？

`InterviewSessionService` 的定位是：

```text
面试会话业务编排服务
```

它不是最底层数据库服务，也不是单纯的 AI 调用服务。它负责把多个子服务串起来，完成“面试会话生命周期”。

看它的方法就能看出来：

```java
public InterviewSessionDTO createSession(CreateInterviewRequest request)
public InterviewSessionDTO getSession(String sessionId)
public Optional<InterviewSessionDTO> findUnfinishedSession(Long resumeId)
public Map<String, Object> getCurrentQuestionResponse(String sessionId)
public InterviewQuestionDTO getCurrentQuestion(String sessionId)
public SubmitAnswerResponse submitAnswer(SubmitAnswerRequest request)
public void saveAnswer(SubmitAnswerRequest request)
public void completeInterview(String sessionId)
public InterviewReportDTO generateReport(String sessionId)
```

这些方法都围绕一个核心对象：

```text
面试会话 session
```

所以它的职责是：

```text
创建会话
恢复会话
推进当前题目
保存答案
完成面试
触发异步评估
生成报告
```

### 5. createSession 方法怎么拆？

核心代码逻辑可以翻译成：

```text
创建一场面试：
1. 如果传了 resumeId，而且不是强制创建，先查有没有未完成会话。
2. 有未完成会话就直接返回，避免重复创建。
3. 没有的话生成新的 sessionId。
4. 确定 skillId 和 difficulty。
5. 查询历史题目，避免重复出题。
6. 调用 questionService 生成面试题。
7. 把会话保存到 Redis。
8. 把会话保存到数据库。
9. 返回 InterviewSessionDTO。
```

对应代码：

```java
if (request.resumeId() != null && !Boolean.TRUE.equals(request.forceCreate())) {
    Optional<InterviewSessionDTO> unfinishedOpt = findUnfinishedSession(request.resumeId());
    if (unfinishedOpt.isPresent()) {
        return unfinishedOpt.get();
    }
}
```

这段是业务规则：

```text
同一份简历如果已经有未完成面试，默认不重复创建。
```

```java
String sessionId = UUID.randomUUID().toString().replace("-", "").substring(0, 16);
```

生成一个短会话 ID。

```java
String skillId = request.skillId() != null ? request.skillId() : InterviewDefaults.SKILL_ID;
String difficulty = request.difficulty() != null ? request.difficulty() : InterviewDefaults.DIFFICULTY;
```

如果前端没传技能方向或难度，就使用默认值。

```java
List<HistoricalQuestion> historicalQuestions =
    persistenceService.getHistoricalQuestions(skillId, request.resumeId());
```

从数据库查历史问题，目的是让 AI 出题时避免重复。

```java
List<InterviewQuestionDTO> questions = questionService.generateQuestionsBySkill(...);
```

把“生成题目”这个复杂子任务交给 `InterviewQuestionService`。

```java
sessionCache.saveSession(...);
```

保存到 Redis。Redis 适合存当前会话状态，比如当前第几题、题目列表、临时答案。

```java
persistenceService.saveSession(...);
```

保存到数据库。数据库适合长期保存会话记录，服务重启后还能恢复。

最后：

```java
return new InterviewSessionDTO(...);
```

返回给前端。

### 6. InterviewQuestionService 为什么还能继续进去？

因为 `InterviewQuestionService` 是“生成题目”这个子业务的专门服务。

它的职责不是管理面试会话，而是：

```text
根据技能方向、简历文本、难度、历史题目、JD，生成 InterviewQuestionDTO 列表。
```

它内部又依赖：

```java
private final StructuredOutputInvoker structuredOutputInvoker;
private final InterviewSkillService skillService;
private final LlmProviderRegistry llmProviderRegistry;
private final PromptSanitizer promptSanitizer;
private final ExecutorService questionExecutor;
```

这些依赖继续细分：

```text
StructuredOutputInvoker  负责调用 LLM，并要求模型按结构化格式返回
InterviewSkillService    负责读取/构建面试技能方向
LlmProviderRegistry      负责选择哪个模型供应商
PromptSanitizer          负责清理用户输入，降低 prompt 注入风险
ExecutorService          负责并发生成简历题和方向题
```

所以 Ctrl + 左键还能继续进去是正常的。项目就是这样从粗到细拆的：

```text
InterviewController
  -> InterviewSessionService
       -> InterviewQuestionService
            -> InterviewSkillService
            -> LlmProviderRegistry
            -> StructuredOutputInvoker
       -> AnswerEvaluationService
       -> InterviewPersistenceService
            -> InterviewSessionRepository
            -> InterviewAnswerRepository
       -> InterviewSessionCache
       -> EvaluateStreamProducer
```

你不需要一次性把每个类都背下来。你要判断它属于哪种角色。

### 7. Service 套 Service 的标准判断

不是所有类都应该叫 Service，也不是所有 Service 都应该互相调用。判断标准是：

```text
如果它表达一个业务能力，可以是 Service。
如果它只负责数据库 CRUD，更适合 Repository 或 PersistenceService。
如果它只负责缓存，更适合 Cache。
如果它只负责发消息，更适合 Producer。
如果它只负责配置/模型选择，更适合 Registry。
如果它只负责对象转换，更适合 Mapper。
```

例如：

```text
InterviewSessionService
业务能力：管理面试会话生命周期

InterviewQuestionService
业务能力：生成面试题

InterviewPersistenceService
数据能力：保存/查询面试会话、答案、报告

InterviewSessionCache
缓存能力：Redis 中保存和恢复会话

EvaluateStreamProducer
消息能力：投递评估任务

LlmProviderRegistry
基础设施能力：根据 provider 获取 ChatClient
```

这就是“Service 里面还有 Service”的合理情况：上层 Service 编排流程，下层 Service 承担某个独立子能力。

### 8. 复现这种代码时怎么写？

不要一上来就写所有类。按业务流程拆。

假设你要复现“创建面试会话”，先写流程草图：

```text
createSession(request):
1. 检查是否有未完成会话
2. 生成 sessionId
3. 解析 skillId / difficulty
4. 查历史题目
5. 生成题目
6. 存 Redis
7. 存数据库
8. 返回 DTO
```

然后把每一步分配给组件：

```text
检查未完成会话        InterviewSessionService + persistence/cache
生成 sessionId         InterviewSessionService 自己做
查历史题目             InterviewPersistenceService
生成题目               InterviewQuestionService
存 Redis               InterviewSessionCache
存数据库               InterviewPersistenceService
返回 DTO               InterviewSessionService 组装
```

于是成员变量自然就出来了：

```java
private final InterviewQuestionService questionService;
private final InterviewPersistenceService persistenceService;
private final InterviewSessionCache sessionCache;
```

如果完成面试后要异步评估，还需要：

```java
private final EvaluateStreamProducer evaluateStreamProducer;
```

如果要生成报告，还需要：

```java
private final AnswerEvaluationService evaluationService;
private final LlmProviderRegistry llmProviderRegistry;
```

如果有 JSON 题目列表要保存/恢复，还需要：

```java
private final ObjectMapper objectMapper;
```

所以成员变量不是凭空冒出来的，而是从业务步骤倒推出来的。

### 9. 一个项目的复现标准

你说“学会一个项目的标准是能够复现他的代码”，这个判断是对的。但复现不是逐字默写，而是能复现结构。

你应该能做到：

```text
1. 看到 Controller 方法，知道它应该调用哪个 Service。
2. 看到 DTO 字段，知道要从哪些 Entity/Service 取数据。
3. 看到一个复杂 Service，能拆出它依赖哪些子能力。
4. 知道哪些逻辑应该在 Controller，哪些应该在 Service。
5. 知道什么时候要 Repository、Cache、Producer、Mapper、Registry。
6. 能按“业务步骤 -> 组件职责 -> 成员变量 -> 方法实现”的顺序写代码。
```

最重要的复现公式：

```text
Controller = HTTP 参数 + 调用 Service + 包装返回

上层 Service = 业务流程编排

下层 Service/组件 = 独立子能力

PersistenceService/Repository = 数据库读写

Cache = 临时状态和快速恢复

Producer = 异步任务投递

DTO = 返回给前端的数据结构

Entity = 数据库表结构
```

如果你能按这个公式去读 `ResumeController`、`ResumeHistoryService`、`InterviewSessionService`、`InterviewQuestionService`，你就不是在“跟着 Ctrl + 左键迷路”，而是在沿着系统分层往下追业务实现。
