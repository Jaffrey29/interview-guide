# 多 LLM 路由实战 · 面试速查

> 基于 `interview-guide` 项目的 `LlmProviderRegistry` 设计，浓缩成 10 个高频面试问答 + 核心知识图谱。

---

## 一、5 个设计目标（必背）

| 目标                         | 通俗解释                                      | 解决什么生产问题                        |
| ---------------------------- | --------------------------------------------- | --------------------------------------- |
| **多 Provider 共存**   | 通义/Kimi/DeepSeek/GLM 同时在线，按业务路由   | 容灾、成本优化、功能互补                |
| **同 Provider 多配方** | 同一模型出三种 ChatClient（默认/Plain/Voice） | 出题要纯 JSON，语音要流式，场景互不串扰 |
| **运行时可写**         | REST API 改 Provider，无需重启                | 轮换 Key、切默认模型不用发版            |
| **配置安全**           | API Key 加密落库，明文不入磁盘                | 防 Key 泄露被盗刷                       |
| **降级兜底**           | DB 挂了自动读 YAML，零迁移成本                | 单元测试不用起 DB，老项目平滑升级       |

## 二、核心架构图

```
业务模块（resume / interview / knowledgebase / voice）
                      │
                      ▼
        ┌──────────────────────────────┐
        │   LlmProviderRegistry        │  ← 唯一入口
        │   - getChatClient(id)        │
        │   - getPlainChatClient(id)   │
        │   - getVoiceChatClient(id)   │
        │   - getEmbeddingModel(id)    │
        │   - reload()                 │
        └──────────┬───────────────────┘
                   │ ConcurrentHashMap 缓存
                   ▼
        ProviderSnapshot (record，不可变)
                   │
       ┌───────────┴───────────┐
       ▼                       ▼
 LlmProviderRepository    LlmProviderProperties
   (DB，加密存储)           (YAML 兜底)
       ▲
       │ AES/GCM/NoPadding
       │ 12 字节随机 nonce
   ApiKeyEncryptionService
```

**一句话**：所有指向大模型的箭头都从 Registry 出发——这就是"收口"。

## 三、三类 ChatClient 配方对比

| 客户端                     | 工具调用 | 记忆           | 流式        | 典型场景                |
| -------------------------- | -------- | -------------- | ----------- | ----------------------- |
| **默认 ChatClient**  | ✅       | ✅             | 按配置      | 知识库问答、RAG         |
| **Plain ChatClient** | ❌       | ❌             | ❌          | 出题、评分（要纯 JSON） |
| **Voice ChatClient** | ✅       | ❌（手动管理） | ✅ 强制流式 | 语音面试                |

**踩坑记录**：早期用统一 ChatClient 出题，模型偶尔触发工具调用、返回带 `tool_call` 字段的混合消息，`BeanOutputConverter` 解析 JSON 直接挂掉。**结构化输出场景下，工具调用就是噪声。**

## 四、缓存 Key 设计的精髓（二级缓存）

```java
// 上层：三种配方各一个 ChatClient 实例
private final Map<String, ChatClient> clientCache = new ConcurrentHashMap<>();
// 下层：同一个 Provider 共用一个 OpenAiChatModel（commit 4ff97b6 引入）
private final Map<String, OpenAiChatModel> chatModelCache = new ConcurrentHashMap<>();

public ChatClient getChatClient(String id)      { return clientCache.computeIfAbsent(id,           k -> createChatClient(id));      }
public ChatClient getPlainChatClient(String id) { return clientCache.computeIfAbsent(id + ":plain", k -> createPlainChatClient(id));}
public ChatClient getVoiceChatClient(String id) { return clientCache.computeIfAbsent(id + ":voice", k -> createVoiceChatClient(id));}

// 三个 create*ChatClient 都走这个方法，共享同一个 ChatModel
private OpenAiChatModel getChatModel(String providerId) {
    return chatModelCache.computeIfAbsent(providerId, id -> buildChatModel(id));
}
```

**两层 Key 的真正含义**：
- **上层 Key**：`dashscope` / `dashscope:plain` / `dashscope:voice` —— 三个独立 ChatClient（Advisor 链不同）
- **下层 Key**：`dashscope` —— 一个 OpenAiChatModel（同一 HTTP 连接池）

**踩坑历史**：早期版本三个 `create*ChatClient` 各自调 `buildChatModel()`，导致同一 Provider 重复构造 3 次 ChatModel + 3 个 HTTP 连接池。commit `4ff97b6` 引入 `chatModelCache` 修复——**Advisor 链隔离 + 底层资源共享**，这才是教科书级设计。

## 五、DB 优先 + YAML 兜底（双写策略）

```java
private ProviderSnapshot loadProviderOrThrow(String providerId) {
    if (providerRepository == null) {
        return loadProviderFromPropertiesOrThrow(providerId); // DB 不可用 → YAML 兜底
    }
    LlmProviderEntity entity = providerRepository.findById(providerId)
        .filter(LlmProviderEntity::isEnabled)
        .orElseThrow(() -> new IllegalArgumentException("Unknown provider: " + providerId));
    return new ProviderSnapshot(
        entity.getId(),
        entity.getBaseUrl(),
        encryptionService.decrypt(entity.getApiKeyNonce(), entity.getApiKeyCiphertext()), // 解密
        entity.getModel(), ...
    );
}
```

**启动时自动播种**（只在第一次跑）：

```java
@PostConstruct @Transactional
public void seedProvidersIfNecessary() {
    if (providerRepository.count() == 0) {
        seedProviders(); // YAML 写进 DB
    }
}
```

**关键**：`if (count() == 0)` 保证只在首启迁移，避免每次启动覆盖运维改的配置。

## 六、API Key 加密（AES-GCM）

```java
public EncryptedValue encrypt(String plainText) {
    byte[] nonce = new byte[12];
    secureRandom.nextBytes(nonce); // 每次新随机 nonce
    Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
    cipher.init(Cipher.ENCRYPT_MODE, secretKey, new GCMParameterSpec(128, nonce));
    byte[] ciphertext = cipher.doFinal(plainText.getBytes(UTF_8));
    return new EncryptedValue(
        Base64.encode(nonce),       // 存 api_key_nonce
        Base64.encode(ciphertext)   // 存 api_key_ciphertext
    );
}
```

**三个生产级细节**：

- **每次新 nonce**：GCM 模式 nonce 重用 = 明文泄露
- **密钥派生两手准备**：合法 Base64（32 字节）直接用；否则走 SHA-256 派生（让运维可以写 `interview-guide-prod-2025` 这种人类可读字符串）
- **双开关安全策略**（最新代码）：
  - `require-encryption-key`：默认 `true`，未配置 `APP_AI_CONFIG_ENCRYPTION_KEY` 直接启动失败
  - `allow-fallback-encryption-key`：仅当前者为 `false` 且这项为 `true` 时，才允许用内置 dev fallback 启动——**双开关防误开**

## 七、热更新机制（reload）

```java
@Transactional
public void updateProvider(String id, UpdateProviderRequest req) {
    rwLock.writeLock().lock();
    try {
        LlmProviderEntity p = getProviderEntityOrThrow(id);
        // 更新字段...
        if (req.getApiKey() != null) {
            EncryptedValue enc = encryptionService.encrypt(req.getApiKey());
            p.setApiKeyNonce(enc.nonce());
            p.setApiKeyCiphertext(enc.ciphertext());
        }
        providerRepository.save(p);
        registry.reload(); // 关键
    } finally {
        rwLock.writeLock().unlock();
    }
}

public void reload() {
    int size = clientCache.size() + chatModelCache.size() + embeddingModelCache.size();
    clientCache.clear();           // 上层 ChatClient
    chatModelCache.clear();        // 下层 OpenAiChatModel（必须清，否则改 baseUrl 不生效）
    embeddingModelCache.clear();
    log.info("[LlmProviderRegistry] Cache cleared ({} entries).", size);
}
```

**为什么不精细化清理某个 ID？** 默认 Provider 切换、Advisor 配置变更、Embedding 默认变更等场景影响面是全局的，精细化容易漏。**缓存撑死十几个对象，重建几乎零成本——简单粗暴最稳**。

**软切换语义**：`ConcurrentHashMap.clear()` 不影响已持有引用的线程，只影响后续 `computeIfAbsent` 重建——天然友好。

## 八、两个低估的细节

### 1. Embedding 防呆校验

把 `qwen-turbo` 当 Embedding 模型用 → 反序列化失败 → 知识库全 FAILED。

```java
private boolean looksLikeChatModel(String model) {
    String lower = model.toLowerCase();
    return lower.startsWith("glm-") || lower.startsWith("deepseek")
        || lower.startsWith("kimi") || lower.startsWith("moonshot")
        || lower.startsWith("qwen") || lower.startsWith("ernie");
}
```

**多层校验**：在 `LlmProviderConfigService` 的创建、更新、连通性测试三个入口都调 `validateEmbeddingConfig`；构造 EmbeddingModel 时 `LlmProviderRegistry.createEmbeddingModel()` 再校验一次——**校验要早，宁可 API 层抛 400，也别让错误配置在向量化任务里悄悄爆炸**。

### 2. baseUrl 兼容（ApiPathResolver）

各厂商 baseUrl 是否带 `/v1` 完全没规律：

- DashScope：`.../compatible-mode/v1` ✅ 带
- DeepSeek：`api.deepseek.com` ❌ 不带
- GLM：`.../api/coding/paas/v4` ✅ 带 `/v4`

```java
private static final Pattern TRAILING_VERSION = Pattern.compile("/v\\d+[a-zA-Z0-9]*$");

public static String resolveVersionedBaseUrl(String baseUrl) {
    String stripped = stripTrailingSlashes(baseUrl);
    if (baseUrlContainsVersion(stripped)) {
        return stripped;          // 已带 /v1 或 /v4，原样返回
    }
    return stripped + "/v1";      // 没带版本号，自动补 /v1
}
```

**正则 `/v\d+[a-zA-Z0-9]*$`** 匹配 `/v1` `/v2` `/v4` `/v1beta` 等所有"语义版本"路径段。

**避免的坑**：如果直接把 DashScope 的 `.../compatible-mode/v1` 拼到 OpenAI 客户端，客户端默认会再补 `/v1/chat/completions`，结果变成 `.../compatible-mode/v1/v1/chat/completions` → 404 让你查半天。

## 十、面试官 8 问（必背答案）

### Q1：为什么同一个 Provider 要分三种 ChatClient？

**核心答**：不同业务场景 Advisor 配方需求不同。

- Plain 出题/评分需要纯 JSON，工具调用会污染输出
- Voice 语音面试需要流式 + 手动 Memory（避免会话串扰）
- 默认场景才需要完整工具+记忆链

**踩坑佐证**：早期统一用一个 ChatClient 出题，模型偶尔触发工具调用导致 `BeanOutputConverter` 解析失败。

---

### Q2：API Key 加密为什么每次生成新 nonce？

**核心答**：AES-GCM 模式下 nonce 重用 = **明文泄露**。GCM 是流密码模式，相同 nonce 加密两段不同明文，攻击者可异或推导出明文异或值。所以必须用 `SecureRandom` 每次生成新 12 字节随机数，和密文一起存库。

**对比**：CBC 模式重用 IV 只是降低安全性，GCM 重用 nonce 是直接破防。

---

### Q3：reload 为什么直接 clear 全部缓存而不精细清理？

**核心答**：

1. **影响面广**：默认 Provider 切换、Advisor 配置变更、Embedding 默认变更等都是全局影响，精细化容易漏
2. **成本低**：缓存对象就十几个，重建几乎零成本
3. **软切换语义**：`ConcurrentHashMap.clear()` 不影响已持有引用的线程，只影响新请求

**简单粗暴反而最稳**——这是工程经验比"教科书最优解"重要的典型例子。

---

### Q4：为什么用 DB + YAML 双写策略，而不是只选一个？

**核心答**：单一方案各有死穴：

- 纯 YAML：改完要重启，多用户协作难，API Key 没法加密
- 纯 DB：单元测试要起 DB，DB 挂了整个应用废

**双写解法**：DB 优先 + YAML 兜底 + 启动时双向同步。

- `providerRepository == null` 时降级读 YAML（典型场景：单测）
- 第一次启动 `count() == 0` 时从 YAML 播种到 DB，之后 DB 为准
- 老用户零迁移成本，运维可在网页改配置

---

### Q5：缓存 Key 为什么设计成 `id` / `id:plain` / `id:voice`？

**核心答**：扁平化 Key + 二级缓存——上层 ChatClient 隔离配方，下层 ChatModel 共享资源。

- 上层 `clientCache`：`dashscope` / `dashscope:plain` / `dashscope:voice` 各一份 ChatClient（Advisor 链不同）
- 下层 `chatModelCache`：`dashscope` 只一份 OpenAiChatModel（HTTP 连接池共享）

**Bug 修复故事**：早期版本三个 `create*ChatClient` 各自调 `buildChatModel()`，同一 Provider 重复构造 3 次底层模型。commit `4ff97b6` 引入 `chatModelCache` 修复——这是阅读源码能学到的活案例：**文档说"共享 ChatModel"，但 2026-05 之前代码并没真共享**。

---

### Q6：如果运维改了 baseUrl，前端发现还是用旧地址，可能问题在哪？

**核心答**：缓存没全清干净。`reload()` 必须同时清三个 Map：`clientCache` / `chatModelCache` / `embeddingModelCache`，漏一个都会导致旧实例残留。

**排查路径**：
1. 看日志确认 `[LlmProviderRegistry] Cache cleared (N entries)` 是否触发
2. 确认 DB 里 baseUrl 已更新（直查 `llm_provider_config` 表）
3. 检查 `reload()` 是不是只 clear 了 `clientCache`、忘了 `chatModelCache`（早期 bug 模式）
4. OpenAI 官方 SDK 内部还可能持有 HTTP 连接池，必要时显式 close 旧 OpenAIClient

---

### Q7：要给某个 Provider 单独加限流，加在哪一层？

**最佳答**：在 `buildChatModel()` 时把 OpenAIClient 包一层代理（或自定义 HTTP 拦截器），**统一拦截 HTTP 请求**。最新代码 `ApiPathResolver.buildOpenAiClient()` 用的是 OpenAI 官方 SDK + `SpringAiOpenAiHttpClient`，可以在 HttpClient builder 里挂拦截器。

**为什么不在 Service 层加**：三种 ChatClient 都会调，业务代码会重复写限流逻辑，新增配方时容易漏。

**配合本项目 `@RateLimit` 注解**：业务侧粗粒度限流（按用户/IP），Provider 侧细粒度限流（按厂商 QPS 配额），双层防护。

### Q8：共用 ChatModel 的好处（4 点）+ 不共用的缺点

**4 个好处**：

1. **节省 HTTP 连接池**：每个 OpenAIClient 持有独立连接池。不共用 = 3N 个连接池，共用 = N 个，省 66%
2. **节省内存**：ChatModel 内含 ToolCallingManager / ObservationRegistry / ChatOptions，N 个 Provider 省 N 份
3. **配置/状态一致**：Token 计数、Observation 指标统一聚合，不会三份分散；连接断开恢复行为一致
4. **热更新原子性**：`reload()` 后三种配方共指向同一个新 ChatModel，不会出现"Plain 用旧 baseUrl、Voice 用新 baseUrl"中间态

**不共用的缺点对照表**：

| 维度                          | 不共用                                                        | 共用                       |
| ----------------------------- | ------------------------------------------------------------- | -------------------------- |
| **首启冷启动**          | 三种配方各 build 一次，构建 3 次 ChatModel 和 3 个连接池      | 只构建 1 次                |
| **HTTP 连接池**         | 3 倍连接数，可能触发 OS 端口/文件描述符上限                   | 1 份连接池                 |
| **内存**                | 3 份 OpenAIClient/ToolCallingManager                          | 1 份                       |
| **Token 用量监控**      | 三套独立指标，要手动聚合                                      | 一套指标                   |
| **修改 baseUrl/apiKey** | reload 后三套都重建，存在并发窗口期看到不一致                 | 三种配方共指向新 ChatModel |
| **未来加全局拦截器**    | 要在 3 个 build 处都加                                        | 改 1 处                    |
| **加全局限流**          | 3 个 HTTP 客户端各走各的，QPS 难以统一                        | 1 个统一拦截               |

**修复启示**（commit `4ff97b6`）：
- 文档说"共享 ChatModel"，但 2026-05 之前代码并没真共享——**文档与代码可能脱节**
- 修复模式很经典：**多级缓存 + computeIfAbsent + reload 同步清理三个 Map**
- 阅读源码能看出"实现与设计意图的偏差"——这是高级开发能力

## 十一、一句话核心总结

> **多 LLM 路由的本质，是把"大模型"从"写死的配置"升级为"可运行时管理的资源"。**
>
> 核心四件事：**Registry 收口** + **三种配方分离** + **DB/YAML 双写兜底** + **热更新清缓存**。
>
> 这套设计的价值不在代码复杂度，而在于解决了生产环境的四个真实痛点：**容灾、多场景、动态切换、安全合规**。
